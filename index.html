02

1. Executive Summary & Problem Definition

HobbyFi connects local hobby communities with venue owners. Our vendor portal lets business operators (badminton clubs, yoga studios, dance academies) manage memberships, bookings, and revenue through a CRM dashboard. The HobbyFi Copilot adds a conversational AI layer on top of this CRM.

The Copilot is designed to address two main patterns of user request:

  1. Read/Analytical Queries: Retrieve business KPIs in natural language (e.g., "what is the revenue of today?", "list trial users of badminton game").
  2. Write/Mutative Actions: Administrative changes that only execute upon explicit vendor approval (e.g., "extend Sarah's free trial by 7 days").

Design Constraints

  • Multi-tenancy isolation: Vendor A must never see Vendor B's data, even through adversarial prompts.
  • Human-in-the-Loop (HITL): Every write operation must generate a preview, pause execution, and wait for manual Approve/Reject.
  • Audit trail: All Copilot actions — reads, writes, approvals, rejections, blocked attempts — must be logged for compliance.
  • Low latency: Vendors expect sub-3s responses for analytical queries.

This document describes an enterprise-ready architecture using Mastra as the central orchestration framework, backed by a robust memory strategy, dual-layer guardrails, and structured approval workflows.

2. Architecture Overview

The HobbyFi Copilot uses a Gateway-Router-Worker architecture pattern. Rather than relying on a single monolithic LLM prompt to handle both reasoning and actions, we separate the responsibilities into specialized agents orchestrated by Mastra Workflows.

Component Description

  • Vendor Portal UI: The chat interface embedded in the vendor CRM. It passes the user query along with session context (e.g., vendor_id, user_role).
  • Input Guardrails: Validates the prompt for injection attacks, PII, and validates that the query refers only to the authenticated vendor's data (isolation check).
  • Mastra Agent Router: A lightweight router (using Claude 3.5 Sonnet or Gemini 2.0 Flash) that reads the query and routes it to the correct specialized worker.
  • Read Worker Agent: Has access to query tools (SQL generators, vector search indexes) and operates under a read-only database connection.
  • Write Worker Agent: Generates proposed state changes, formatting them into a standard execution schema. It cannot commit directly; instead, it outputs an execution payload and registers a transaction as PENDING.
  • Approval System (HITL): A persistent state machine that holds pending mutations, generates UI-renderable "approval cards" for the vendor, and executes the database update only upon verified HTTP POST approval.

5. Memory Strategy

A great Copilot must recall details like "What was the name of the trial user we talked about just now?" (short-term) and "Which badminton court does the vendor prefer booking?" (long-term).

Memory Layers:

  • Short-term Conversational Memory (Session Cache):
    • Implementation: Redis-backed sliding window of the last 10 messages.
    • Details: Keeps the immediate context of the chat. When a vendor says "increase his free trial", the system references the previous message's subject entity (e.g. user John Doe).
  • Long-term Semantic Memory (Profile Vector Store):
    • Implementation: PostgreSQL pgvector store containing periodic summaries of past chats.
    • Details: A background Mastra workflow runs asynchronously when a session closes. It summarizes the chat, extracts preferences (e.g., "Vendor prefers keeping trial periods under 14 days"), and embeds them using text-embedding-3-small.
  • Entity Memory (Contextual Injection):
    • Implementation: Context payload containing the authenticated vendor profile, active configuration settings, and structural database database metadata. Injected directly into system prompt.

6. Guardrails Framework

Safety is paramount. Operating an AI CRM copilot without guardrails exposes HobbyFi to data leaks (vendor B seeing vendor A's revenue) and prompt injections.

Input Guardrails

  • Vendor Context Locking (Multitenancy Enforcement): Every tool execution has access to the current_vendor_id via the request context. The database queries always hardcode the WHERE vendor_id = :current_vendor_id clause. Even if the LLM attempts to query another vendor's ID, the tool layer rejects it.
  • Prompt Injection Detection: A fast LLM check (using Gemini 2.0 Flash) inspects the prompt for phrases like "ignore previous instructions", "export all users", or system overrides.
  • PII Masking: Regular expressions mask credit cards, passwords, and phones before passing prompts to the external LLM provider.

Output Guardrails

  • SQL Safe Execution: No raw SQL generated by the LLM is executed directly against the production database. The LLM must output structured JSON payloads representing parameters, which are then passed to parameterized database queries.
  • Sensitive Data Filter: Assures that passwords or system internals never leak in the assistant responses.

7. Workflow Orchestration (Mastra Workflows)

For write operations, we implement a multi-step Mastra Workflow containing conditional gates and human-in-the-loop validation.

Detailed Mutation Flow Example: "Increase free trial of John Doe by 5 days"

  1. Step 1: Parse & Validate
    • Tool: ResolveUserByName(name: "John Doe") ➔ Returns User Object {id: "usr_992", trial_days: 3}.
    • Guardrail check: Is user "usr_992" owned by active Vendor? Yes.
  2. Step 2: Generate Mutation Request
    • Tool: CraftTrialExtensionMutation(user_id: "usr_992", extend_days: 5).
    • Output: Mutation Payload: { action: "UPDATE_USER", fields: { trial_days_remaining: 8 }, original: 3 }.
    • Save to Redis: KEY pending:vendor_123:mut_abc ➔ Expires in 15 mins.
  3. Step 3: Await Vendor Approval (Gated Step)
    • Server suspends workflow execution.
    • Sends UI instruction payload: { status: "REQUIRES_APPROVAL", approval_id: "mut_abc", type: "TRIAL_EXTENSION", summary: "Extend John Doe's trial from 3 to 8 days." }.
    • Chat renders custom interactive button card.
  4. Step 4: Execute on Event Action
    • Vendor clicks "Approve". UI sends POST /api/copilot/approve { approval_id: "mut_abc" }.
    • Backend resumes Mastra workflow, verifies cryptographic token.
    • Tool: CommitDBUpdate(approval_id: "mut_abc") ➔ Executes database transaction.
    • UI Updates state, chat responds: "Trial extended successfully. John Doe now has 8 days remaining."

By leveraging this architectural blueprint, the HobbyFi Copilot remains secure, accurate, and completely under the control of the business owners, preventing unauthorized changes while delivering the efficiency of conversational AI.
